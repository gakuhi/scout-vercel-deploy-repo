プロダクト KPI（DAU / WAU / MAU）の定義と集計方針を決定する。SLO ではなく **プロダクト健全性指標** として扱う（アラート対象外）。

---

## 1. アクティブユーザーの定義

「ログインしただけ」では不十分。**何らかの主要アクションを実施したユーザー** をアクティブ定義する。

### 1.1 学生のアクティブ定義

以下のいずれかを **同日（JST）** に行ったユーザーをアクティブとする。

| アクション | 取得元 |
| :---- | :---- |
| ログイン（成功） | Supabase Auth ログ |
| スカウト閲覧（受信スカウトの個別ページ表示） | アプリ層構造化ログ |
| チャット送信 | アプリ層構造化ログ（`stage: chat-send`） |
| プロフィール編集（保存） | アプリ層構造化ログ |
| イベント詳細閲覧 | アプリ層構造化ログ |

### 1.2 企業のアクティブ定義

以下のいずれかを **同日（JST）** に行ったメンバーをアクティブとする（メンバー単位）。

| アクション | 取得元 |
| :---- | :---- |
| ログイン（成功） | Supabase Auth ログ |
| 学生検索（`/company/students` で検索クエリ実行） | アプリ層構造化ログ |
| スカウト送信 | アプリ層構造化ログ（`03-slo-scout-failure.md` の ① と同じ） |
| チャット送信 | アプリ層構造化ログ |
| 求人 / イベントの作成・編集 | アプリ層構造化ログ |

### 1.3 重複排除

| 単位 | 定義 |
| :---- | :---- |
| **学生 DAU** | 該当日にアクティブ条件を満たした **distinct student_id** の数 |
| **学生 WAU** | 該当週（月〜日 JST）に少なくとも 1 日アクティブだった **distinct student_id** |
| **学生 MAU** | 該当月（1 日〜末日 JST）に少なくとも 1 日アクティブだった **distinct student_id** |
| **企業 DAU/WAU/MAU** | 同様に **distinct company_member_id**（企業単位ではなくメンバー単位）|

> 企業を「企業単位」と「メンバー単位」両方で集計する余地あり。MVP は **メンバー単位** に統一し、企業単位は派生指標として後追い実装する。

## 2. 集計元の選定

候補: GA4 / Supabase ログ / Axiom / アプリ DB。

| 集計元 | 長所 | 短所 | 採用判断 |
| :---- | :---- | :---- | :---- |
| **GA4** | 既存導入済み（前提）、ページビュー軽量 | ログイン状態とユーザー ID の紐付けが要 user_id 設定。プライバシー制約 | **補助**: 流入分析・LP 計測で利用 |
| **Supabase Auth ログ** | ログイン履歴は確定的 | アクション系の取得不可 | **一部利用**: ログイン判定のみ |
| **Axiom**（構造化ログ） | アクション粒度・リアルタイム性・SQL 風クエリ | ログ送信実装が前提 | **メイン**: アクティブ判定の主集計元 |
| **アプリ DB（直接 SELECT）** | 確定的 | 重い集計が本番 DB を圧迫 | **不採用**: 集計用に独立した経路を持つ方が良い |

### 2.1 採用構成

**Axiom をメインの集計元** とし、Supabase Auth ログを Axiom に転送（Log Drain）して統合する。GA4 は流入分析・LP 計測の補助。

```
[アプリ層 構造化ログ]──┐
                      ├──> Axiom ──> ダッシュボード / 月次サマリ
[Supabase Auth Logs]──┘

[GA4]──> LP・流入分析（補助、KPI には使わない）
```

> 構造化ログには **`event_type`, `user_id` (または student_id / company_member_id), `occurred_at`** を必須フィールドとして含める（実装は #45）。

## 3. 集計頻度・公開先

### 3.1 集計頻度

| 指標 | 集計頻度 | 反映ラグ |
| :---- | :---- | :---- |
| DAU | 日次（JST 翌 0:30） | 30 分以内 |
| WAU | 日次（JST 翌 0:30、ローリング 7 日） | 30 分以内 |
| MAU | 日次（JST 翌 0:30、ローリング 30 日 / 暦月） | 30 分以内 |
| Sticky factor（DAU / MAU） | 日次 | 上記に追従 |

> リアルタイム集計は不要。1 日 1 回の Axiom Monitor / Cron で十分。

### 3.2 ダッシュボード公開先

| 公開先 | 用途 | アクセス権 |
| :---- | :---- | :---- |
| **Axiom Dashboard `kpi-active-users`**（新規作成） | チーム内向け。リアルタイムに近い数値確認 | チーム全員（閲覧）|
| **Notion or Google Sheet（週次サマリ）** | 経営判断・週次定例向け | チーム全員 |
| **GA4 ダッシュボード** | 流入経路の補助指標 | チーム全員 |

実装は #321（マーケモニタリング基盤セットアップ）で。本ドキュメントは **定義** に専念。

## 4. ダッシュボードに置くべきパネル

| パネル | 内容 |
| :---- | :---- |
| 学生 DAU 推移 | 直近 30 日の折れ線 |
| 学生 WAU / MAU 推移 | 直近 12 週・直近 6 ヶ月 |
| 企業 DAU / WAU / MAU 推移 | 同上 |
| Sticky factor（DAU/MAU） | 学生・企業それぞれ |
| アクション別アクティブ構成比 | スカウト閲覧 / チャット / プロフィール編集 等 |
| 新規 vs リピート | 学生・企業 |

---

## 5. プライバシー・個人情報配慮

| 観点 | 対応 |
| :---- | :---- |
| 個人特定情報のログ送出禁止 | `02-security-requirements.md` 6.2 に従い、メールアドレス・氏名等は送出しない |
| user_id の扱い | Supabase の `auth.uid()`（UUID）のみログに含める。集計時は集約済み数値のみダッシュボードに表示 |
| GA4 の user_id 設定 | ハッシュ化した値で連携（実装は #321） |
