LINE ログイン・マジックリンク・企業パスワードログイン の失敗率を監視するための目標値と、不正アクセス疑いとユーザーミスを切り分ける方針を定義する。

---

## 1. 監視対象

| 認証種別 | 主な失敗ケース | 取得元 |
| :---- | :---- | :---- |
| **LINE ログイン**（学生） | OAuth state 不一致 / token 取得失敗 / コールバック中の DB エラー / 既存メールとの衝突 | アプリ層構造化ログ → Axiom |
| **マジックリンク**（学生） | リンク失効 / 既使用 / メール未着でユーザーが諦め | Supabase Auth Logs + アプリ層 |
| **企業パスワードログイン** | 誤パスワード / 未確認メール / アカウントロック | Supabase Auth Logs + アプリ層 |
| **企業パスワードリセット** | 期限切れ / トークン誤り | Supabase Auth Logs |
| **MFA（TOTP）検証** | コード誤り / 時計ずれ | Supabase Auth Logs |

> Supabase Auth のログは **Axiom に Log Drain 連携**して集計する前提（#45 範囲）。

## 2. 通常時の失敗率レンジ（想定）

実測前の想定値。リリース後に実測で更新する。

| 認証種別 | 想定通常レンジ | 主な失敗要因 |
| :---- | :---- | :---- |
| LINE ログイン | 1〜5% | ユーザーが LINE アプリ未インストール、戻るボタン操作で中断 |
| マジックリンク | 5〜15% | メール未着・遅延、リンク有効期限切れ（24h） |
| 企業パスワード | 5〜10% | 入力ミス（パスワード忘れ、CapsLock 等） |
| 企業パスワードリセット | 5〜15% | リンク有効期限切れ（1h） |
| MFA 検証 | 1〜3% | TOTP の時計ずれ・コード入力ミス |

これらは「ユーザーミス起因の通常失敗」であり、SLO 違反扱いはしない。**異常時の発火条件**（3 章）でしきい値を超えた場合のみアラート対象。

---

## 3. アラート発火条件（異常検知）

通常レンジを超えた場合と、不正アクセス疑いを切り分けてアラートを設計する。

### 3.1 サービス健全性ベース（広域影響）

該当する認証フロー全体で異常な失敗率が発生 → サービス側の不具合を示唆。

| 認証種別 | Severity | 発火条件 | 想定原因 |
| :---- | :---- | :---- | :---- |
| LINE ログイン | **P1** | 失敗率 **30% を 10 分継続** | LINE Login 障害 / コールバック route の不具合 |
| LINE ログイン | P2 | 失敗率 **15% を 1 時間継続** | 軽微な不具合・LINE 側の遅延 |
| マジックリンク | **P1** | 失敗率 **40% を 30 分継続** | メール配信障害（#318 と相関）/ リンク生成バグ |
| マジックリンク | P2 | 失敗率 **25% を 1 時間継続** | |
| 企業パスワード | P2 | 失敗率 **30% を 1 時間継続** | アカウントロック大量発生・パスワードポリシー変更時の混乱 |
| MFA 検証 | P2 | 失敗率 **20% を 1 時間継続** | TOTP の時計ずれ広域発生 |

### 3.2 不正アクセス疑い（攻撃検知）

特定のユーザー・IP・パターンに集中した失敗 → 攻撃の可能性。

| パターン | Severity | 発火条件 | 備考 |
| :---- | :---- | :---- | :---- |
| 同一 IP からの認証失敗 | **P1** | 10 分間に **同一 IP から 20 回以上失敗** | クレデンシャルスタッフィング疑い |
| 同一メール宛の失敗 | **P1** | 10 分間に **同一メールに 10 回以上失敗** | 標的型ブルートフォース疑い |
| 連続パスワードリセット要求 | P2 | 1 時間に **同一メールから 5 回以上のリセット要求** | 攻撃者によるリセット悪用 |
| 多数の異なるメールへの失敗 | P2 | 5 分間に **同一 IP から 50 種以上の異なるメール宛の失敗** | アカウント列挙攻撃疑い |

### 3.3 ユーザーミスとの切り分け方針

| 観点 | ユーザーミス（SLO 対象外） | 不正疑い（即対応） |
| :---- | :---- | :---- |
| 試行回数 | 1〜3 回で諦め or 成功 | 短時間に多数 |
| IP / アカウント分散 | 単一 IP × 単一アカウント | 単一 IP × 複数アカウント、または逆 |
| User-Agent | 通常ブラウザ | スクリプト的（curl/python-requests 等） |
| 時刻パターン | 散発的 | 機械的・等間隔 |

切り分けは **ログのフィールドベース集計**（`ip`, `email_hash`, `user_agent`, `failure_reason`）で実施。詳細条件は #46 でクエリ化。

## 4. アラート通知先

`03-slo-error-rate.md` 4 章と同一基準。

| Severity | 一次通知 | 二次通知 |
| :---- | :---- | :---- |
| P1 | Slack `#alerts-prod` に **@福田** メンション | 5 分以内未確認でメール |
| P2 | Slack `#alerts-prod` のみ | なし |

不正アクセス疑い（3.2）の P1 は、**Slack `#security-alerts`**（要新設）に分離する案も #46 検討時に議論する。

## 5. 既存対策との関係

`02-security-requirements.md` 1.3（ブルートフォース対策）との関係を明確化。

| 既存対策 | 本ドキュメントとの関係 |
| :---- | :---- |
| Supabase Auth レートリミット（1 分 10 回 / IP） | 第一線防御。本ドキュメントの異常検知（3.2）はそれをすり抜けた長期パターンを補足 |
| アカウントロック（5 回失敗で 15 分） | ロック発生数自体を別途モニタする |

## 6. 計測実装メモ（参考）

アプリ層の構造化ログには以下のフィールドを統一して出力する想定（実装は #45）:

```ts
// LINE ログイン callback の失敗ログ例（疑似コード。実装は #45）
{
  level: "warn",
  component: "auth",
  flow: "line-login-callback",
  status: "failed",
  failure_reason: "state-mismatch",  // state-mismatch / token-error / db-error / email-conflict
  ip_hash: "...",
  email_hash: "...",
  user_agent: "...",
  request_id: "..."
}
```

メールアドレス・IP は **ハッシュ化**してログに保存（`02-security-requirements.md` 6.2 のログに含めない情報原則に整合）。
