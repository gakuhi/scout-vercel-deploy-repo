# スカウトサービス データ連携方式の検討 — フィードバック依頼

## プロダクト概要

既存4プロダクト（面接練習AI、スマートES、企業分析AI、すごい就活）の就活生データを統合し、企業が学生をスカウトできるプラットフォームを構築する。月間新規登録者約6,000人の既存ユーザーベースを活用する。

## 技術スタック

- スカウトサービス本体: Next.js (App Router) + Supabase + Vercel
- AIマッチング: Claude API

## 連携対象プロダクトのDB基盤

| プロダクト | DB基盤 | 備考 |
|---|---|---|
| 面接練習AI | Supabase (PostgreSQL) | |
| 企業分析AI | Supabase (PostgreSQL) | |
| スマートES | PlanetScale (MySQL) | Scaler Pro プラン |
| すごい就活 | Bubble | Bubble Data APIのみ、リアルタイム通知手段なし |

## ユーザー突合方式

メールアドレスで突合する。誤連携リスク（typoアドレスの第三者取得等）への対策として、データ連携同意フロー内で連携データのプレビュー確認ステップを設ける。

## 同意フロー

全ユーザーのデータを同意有無にかかわらず事前に同期しておく。同意前のデータは企業に開示しない（RLSで `data_consent_granted_at IS NOT NULL` の学生のみ企業が閲覧可能）。同意の瞬間にデータ取得を待つ必要がないため、UXが良い。

## 取得するデータ

各プロダクトから必要最小限のデータのみ取得する。

- **面接練習AI**: 評価スコア3種（論理構成・質疑応答・回答内容）、強み、改善点、練習業界。面接の全文書き起こし（transcripts）は取得しない（1人最大~42,000トークン、evaluation_dataで代替可能）
- **企業分析AI**: 調べた企業名・業界リスト、ユーザーの質問傾向。全チャット履歴は取得しない（1人最大~90,000トークン）
- **スマートES**: ES本文、志望動機、志望業界
- **すごい就活**: 診断結果、SPI模試スコア、履歴書要約

## AIマッチングの仕組み

ベクトル検索は使わない（Claude APIにEmbedding APIがない、MVP規模でベクトル検索の恩恵が薄い、プロンプト調整だけで評価基準を変えられるのが強み）。

1. 各プロダクトのデータからClaude APIで統合プロフィールを事前生成（1人~4,500トークン入力 → ~500トークンの要約として保存）
2. マッチング時はハードフィルター（卒年、文理、勤務地）で候補を絞り、統合プロフィールだけをClaudeに投げてスコア+理由を生成

月間コスト試算: 統合プロフィール生成 ~$90 + マッチング ~$30 = **約$120/月**（10,000人規模）

## 検討した連携方式

以下の方式を検討し、最終的に**方式F（read-onlyロール + ETL）**を有力候補としている。

### 方式A: ETL（事前同期）

各プロダクトのDBからデータを取得し、スカウトDBの synced_* テーブルにコピー。Supabase Realtime/Database Webhook、PlanetScale Connect(CDC)、Vercel Cron等で同期。

- ○ スカウトDBだけで完結、外部DB障害に影響されない
- ○ スカウトDBのRLSが適用される
- × 同期ジョブの実装・監視・リトライが必要
- × 各プロダクトのDB認証情報（service_roleキー等）が必要 → 漏洩時に全データ流出リスク

### 方式B: 直接クエリ（service_roleキー）

スカウトサーバーが各プロダクトDBにservice_roleキーで直接SELECT。synced_*テーブル不要。

- ○ 実装が最もシンプル
- ○ 常に最新データ
- × service_roleキーはRLSバイパス。漏洩=全データ読み書き可能
- × 各プロダクト側が公開範囲をコントロールできない
- × スカウトDBにデータがないためRLSでのアクセス制御がコード依存になる

### 方式C: 直接クエリ + read-onlyロール + View

各プロダクト側で連携用Viewとread-onlyロールを作成。スカウトサーバーはViewのみ参照可能。

- ○ 不要データ（transcripts等）に物理的にアクセス不可
- ○ チーム間の責任分界点が明確
- × PlanetScaleはGRANT/REVOKE非対応で適用不可
- × 直接クエリのためスカウトDBのRLSが効かない

### 方式D: 各プロダクト側にAPI層を置く

各プロダクトがスカウト連携用のREST APIを公開。

- ○ 最もセキュア（DB認証情報を一切共有しない）
- ○ 各プロダクト側が公開範囲を完全にコントロール
- × 3プロダクト分のAPI実装が必要（チームリソース的に厳しい）

### 方式E: Read Replica

Supabaseの読み取り専用レプリカを使用。

- ○ 物理的に書き込み不可、本番DBに負荷をかけない
- × 全テーブルが丸見え（テーブル・カラム単位の制限不可）
- × 追加コスト ~$10/月/プロダクト
- × PlanetScale/Bubbleには適用不可

## 有力候補: 方式F（read-onlyロール + ETL）

方式Aと方式Cの良いところを組み合わせたハイブリッド。

```
[同期ジョブ（サーバーサイドのみ、定期実行）]
スカウトサーバー ──read-onlyロール──→ 面接AI DB（SELECTのみ）
スカウトサーバー ──read-onlyロール──→ 企業分析AI DB（SELECTのみ）
スカウトサーバー ──read-onlyパスワード──→ PlanetScale（SELECTのみ）
スカウトサーバー ──Bubble Data API──→ すごい就活

    ↓ 必要なカラムだけ抜いて

スカウトDB の synced_* テーブルに保存

[ユーザーのアクセス]
学生/企業 → スカウトDB（synced_* / 統合プロフィール）
    → スカウトDBのRLSが適用される
    → 各プロダクトDBには一切触れない
```

### 二重の防御

1. **read-onlyロール**: 各プロダクトDBへの書き込みリスクがゼロ。キー漏洩時も読み取りのみ
2. **ETL + スカウトDBのRLS**: データをスカウトDBに入れることで、RLSによるアクセス制御がDBレベルで強制される（「学生は自分のデータだけ」「企業は同意済みの学生だけ」がコードではなくRLSで保証される）

直接クエリ方式（B/C）だとアクセス制御がアプリケーションコード依存になり、バグ1つで他人のデータが見えるリスクがある。ETLでスカウトDBに入れることでRLSがセーフティネットとして機能する。

### 各プロダクトのread-only接続

| プロダクト | 方法 |
|---|---|
| 面接AI (Supabase) | カスタムPostgreSQLロール（SELECT権限のみ）を作成 |
| 企業分析AI (Supabase) | 同上 |
| スマートES (PlanetScale) | ブランチパスワード作成時にRead-onlyロール指定（ダッシュボードで設定） |
| すごい就活 (Bubble) | Bubble Data API（読み取りのみ） |

### 懸念点・未解決事項

1. **同期ジョブの実装・監視コスト**: Vercel Cronで定期実行、差分同期（synced_at以降のデータのみ取得）で軽量化する想定だが、エラーハンドリング・リトライ・監視の実装は必要
2. **データの鮮度**: ポーリング間隔に依存。Supabase Database Webhookを使えばほぼリアルタイムにできるが実装が増える
3. **read-onlyロールでも全テーブルSELECT可能**: 取得するデータの制限はコードで制御。Viewを追加すればDB側でも制限できるが、各プロダクトチームへの依頼が必要
4. **チーム間の責任分界**: service_roleではないが、read-onlyロールでも全テーブルが読める点について各プロダクトチームとの合意が必要
5. **Supabaseのカスタムロール作成にはPostgreSQL直接接続が必要**（Supabase JS クライアント経由では使えない。サーバーサイドの同期ジョブからDB直接接続する）

## フィードバックをもらいたいポイント

1. **方式Fの設計判断に穴はあるか？** 見落としているリスクや、より良い方式はあるか
2. **read-onlyロール + ETL の組み合わせは妥当か？** read-onlyロールだけ、またはETLだけでは不十分と判断した理由は上記の通りだが、この判断は正しいか
3. **同期頻度のバランス**: リアルタイムに近づけるほど実装が複雑になる。MVP段階では15分〜1時間のCronポーリングで十分か
4. **PlanetScale（GRANT非対応）への対応**: Read-onlyパスワード + コードでの取得カラム制御で十分か。それとも別のアプローチが必要か
5. **その他、この構成で気になる点**
