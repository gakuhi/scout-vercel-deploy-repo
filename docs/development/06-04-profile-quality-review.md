# 全体観察（参考）

5人を通じて見えた、設計上検討の余地がある点:

1. **データ源の偏り**: smartes のみ取得のため、interview_ai のスコア（logicalStructure / qaSkill）や comp_ai のリサーチ傾向が `logical_thinking` や `interested_industries` の根拠として使えない。User #3 (caren1224) は sources 上3プロダクト登録だが score_confidence は55どまり

2. **取得件数の上限**: User #5 (fanmac55pq) は93件中22件しかLLMに渡らない。志望動機38件 → 10件抜粋では「量で本気度を示そうとしている学生」のシグナルが正確に伝わらない可能性。逆に件数が極端に多い場合は `regenerated_count` で並び替えて質の高いものを優先する設計が有効か

3. **同一エピソードの重複**: User #3 (caren1224) のガクチカ4件、User #4 (bp23101) のES回答6件など、同じ題材の複数生成が混入。重複検知やまとめが必要かもしれない

4. **`score_confidence` の運用**: 現状の実装では「データ源の数」が主に効くが、**データの質**（推敲回数の分布、エピソードの多様性）を加味する余地がある。User #5 のように「量はあるが薄い」ケースで confidence を盛りすぎないよう注意

5. **`leadership=null` の判定**: User #5 のように組織を率いた経験がないケースで適切に null を返せた。ただしレビュアーには「null は『リーダーシップが低い』ではなく『判定保留』である」点の周知が必要
